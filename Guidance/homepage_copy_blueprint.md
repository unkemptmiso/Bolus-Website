# Bolus Homepage Copy & SEO Blueprint

This blueprint applies the "Human-First" Reddit philosophy. The visible text is ultra-clean, clinical, and punchy to optimize for human conversion. The SEO keywords ("anesthesia record app", "office-based anesthesia charting") are injected using "invisible" methods like semantic HTML and Meta tags.

---

## 0. Invisible SEO (The `<head>` Code)
These elements are read by Google and AI bots, but are invisible on the actual page design.
* **Title Tag (50-60 chars)**: Bolus | Modern Office-Based Anesthesia Record App 
* **Meta Description (150-160 chars)**: Ditch paper charting. Bolus is a clean, mobile-first anesthesia record app designed to help office-based practices create highly defensible documentation.
* **Schema Markup**: `SoftwareApplication` JSON-LD payload injected in the head.

## Section 3: The Solution / Transformation

**Goal**: Introduce Bolus as the practical replacement for paper before we get into the deeper feature walkthrough.

**Section 3 Direction**:
This section should not feel like a feature dump yet. It should answer one simple question:

**If paper is the problem, what actually replaces it?**

The strongest framing here is:
- mobile charting
- one clean workflow
- a record that is easier to stand behind later

### Recommended Direction

* **Section Headline (H2)**:
  * *Charting built for speed. Records made to last.*
* **Support Copy (P)**:
  * Bolus replaces paper with a mobile workflow that is fast during the case and produces a cleaner record when the case is done.
* **Benefit Blocks (H3 + P)**:
  * **Built for the pace of the case.**
    * Chart directly on iPhone or iPad with a workflow that is quick, focused, and easy to use in real time.
  * **One record from start to finish.**
    * Pre-op, intra-op, and post-op stay connected in one place, so the record builds as the case moves.
  * **Cleaner when it matters later.**
    * Export a clear PDF record that is legible, organized, and easier to stand behind.
* **Why this works**:
  * This is the clearest bridge into the next section, where we break down the workflow by pre-op, intra-op, and post-op.
* **Visual**:
  * iPhone and iPad UI shown side by side with a subtle transition toward the final PDF output.
* **Inline CTA**:
  * See the workflow

### Alternate Supporting Copy Options

* **Option A**
  * Bolus replaces paper with a mobile workflow that is fast during the case and produces a cleaner record when the case is done.
* **Option B**
  * Bolus helps you chart quickly in the room and leave with a record that is clearer, more organized, and easier to trust.
* **Option C**
  * Fast charting is only half of it. Bolus also gives you a record that reads clearly when you need it later.

## Homepage Skeleton After Section 3

This is the recommended homepage structure from this point forward. The order is meant to stay logical:

1. problem
2. solution
3. workflow proof
4. trust
5. output
6. pricing
7. FAQ

### Section 4: Workflow Breakdown (Pre-Op / Intra-Op / Post-Op)

**Goal**: This is where the product proves itself. We move from high-level promise into concrete workflow.

* **Section Headline (H2)**:
  * *One workflow from pre-op to post-op.*
* **Support Copy (P)**:
  * Bolus is built to follow the case from start to finish, without sending you back to paper.
* **Structure**:
  * **Pre-Op**
    * Patient details
    * case setup
    * evaluation inputs
    * consent and readiness fields
  * **Intra-Op**
    * timed charting
    * vitals capture
    * medication logging
    * airway and event documentation
  * **Post-Op**
    * recovery tracking
    * discharge criteria
    * final notes and case completion
* **Design Note**:
  * This section can be built as a three-panel sequence, sticky device walkthrough, or horizontal product story.

### Section 5: Security

**Goal**: Reassure without turning the homepage into a compliance lecture.

* **Section Headline (H2)**:
  * *Your records stay protected.*
* **Support Copy (P)**:
  * Security should be clear and boring. Bolus keeps records encrypted and designed for controlled access.
* **Key Points**:
  * encrypted on device
  * designed for privacy
  * no vague enterprise language
  * link out to the deeper security page for full detail

### Section 6: Export

**Goal**: Show the output, because the final record is one of the strongest parts of the product story.

* **Section Headline (H2)**:
  * *The finished record matters.*
* **Support Copy (P)**:
  * Bolus turns charting into a clean PDF export that looks organized, legible, and professional.
* **Key Points**:
  * readable structure
  * clean timestamps
  * polished PDF layout
  * easier storage, retrieval, and review than paper
* **Visual**:
  * final exported record shown clearly, not as a tiny mocked-up detail

### Section 7: Pricing

**Goal**: Keep pricing direct and low-friction.

* **Section Headline (H2)**:
  * *Simple pricing for modern charting.*
* **Support Copy (P)**:
  * Clear plan, clear value, no bloated pricing table unless there is a real need for one.
* **Key Points**:
  * price
  * what is included
  * App Store download CTA
  * optional practice or team note if relevant later

### Section 8: FAQ

**Goal**: Capture SEO depth and answer objections without cluttering the main sections above.

* **Section Headline (H2)**:
  * *Frequently asked questions*
* **Purpose**:
  * answer practical objections
  * support long-tail search intent
  * create room for keywords naturally without forcing them into the higher-conversion sections
* **Suggested FAQ Themes**:
  * What is Bolus?
  * Who is Bolus for?
  * Can I chart on iPhone and iPad?
  * Does Bolus replace paper anesthesia records?
  * How does PDF export work?
  * How does Bolus handle security and privacy?
  * What kind of practices use Bolus?
  * How much does Bolus cost?
