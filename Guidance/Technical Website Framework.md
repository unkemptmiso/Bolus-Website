# Technical Website Framework

This document outlines the technical architecture, infrastructure, and implementation strategy for the Bolus marketing website. 

**Objective:** Build a highly performant, SEO-optimized, and dynamic marketing site capable of scaling into a full-stack web application with user authentication and payment processing, while maintaining the absolute minimum client-side bundle size.

---

## 1. Core Architecture Stack

**1.1. Framework: Astro**
*   **Role:** The primary static site generator (SSG) and orchestrator.
*   **Purpose:** To deliver raw, pre-rendered HTML to the client by default, ensuring lightning-fast load times and optimal SEO. 
*   **Mechanism:** Astro will strip away all unnecessary JavaScript. "Islands Architecture" will be utilized to selectively hydrate interactive components only where absolutely necessary.

**1.2. Interactive Layer: React (TypeScript)**
*   **Role:** Client-side interactivity.
*   **Purpose:** Used strictly for isolated, dynamic components (e.g., interactive pricing toggles, complex case simulations, or authentication forms) via the `@astrojs/react` integration.
*   **Execution:** React components will be mounted using Astro client directives (e.g., `client:load`, `client:visible`) to ensure they only execute JavaScript when they enter the viewport or are immediately needed.

**1.3. Animation Engine: Framer Motion**
*   **Role:** Physics-based animation library.
*   **Purpose:** To handle complex, scroll-triggered DOM manipulations and layout transitions without degrading performance. Must be used inside React "Islands" where animation is required.

---

## 2. Infrastructure & Hosting (Cloudflare Ecosystem)

**2.1. Edge Hosting: Cloudflare Pages**
*   **Deployment:** The main Astro build output (the `dist/` directory) will be deployed to Cloudflare Pages.
*   **Performance:** Assets will be cached at the Cloudflare Edge, ensuring ultra-low latency globally.

**2.2. Serverless Backend: Cloudflare Workers**
*   **Role:** The invisible backend layer bridging static content with dynamic database needs.
*   **Purpose:** Handling securely sensitive operations that cannot occur on the client side.
*   **Integration:** Can be directly integrated using Astro's Cloudflare adapter (`@astrojs/cloudflare`) for hybrid rendering (SSR) if specific pages need runtime execution.

---

## 3. Dynamic Integrations (Auth & Payments)

**3.1. User Authentication (Firebase / Custom)**
*   **Implementation:** Client-side Firebase SDK injection within specific React Islands, or SSR verification via Cloudflare Workers.
*   **State Management:** Auth state should dictate component rendered via conditional hydration. Validated JWT tokens will be passed securely via HttpOnly cookies if utilizing SSR on Cloudflare.

**3.2. Payment Processing (Stripe)**
*   **Security:** Stripe Secret Keys must NEVER be exposed to the client. They will reside in Cloudflare Pages environment variables.
*   **Checkout Flow:** Client-side Stripe.js will be used to redirect users to a Stripe Checkout Session generated by a Cloudflare Worker proxy route.
*   **Webhooks:** A dedicated Cloudflare Worker route (e.g., `/api/stripe-webhook`) will listen for `checkout.session.completed` events from Stripe to trigger database updates (like activating a user's subscription).

---

## 4. Performance & SEO Directives

**4.1. Asset Optimization**
*   **Images:** Utilize Astro's native `<Image />` component to automatically convert heavy assets into `.webp` or `.avif`, enforce lazy-loading (`loading="lazy"`), and prevent Cumulative Layout Shift (CLS) by defining exact heights and widths.
*   **JavaScript Payload:** Strict enforcement of 0kb JS on pure marketing routes.

**4.2. SEO Generation**
*   **Sitemaps:** Use `@astrojs/sitemap` integration to auto-generate `sitemap.xml` on build.
*   **Metadata:** Implement a reusable `<SEO />` Astro component injected into the `<head>` of every layout to dynamically populate title, description, and OpenGraph tags per route. 

---

## 5. Development Workflow & Responsiveness

**5.1. File-Based Routing**
*   Leverage Astro's `src/pages/` directory structure for implicit routing.
*   API endpoints (like the Stripe webhook listener) must be defined in `src/pages/api/` and exported as standard HTTP methods (e.g., `export const POST = ...`).

**5.2. Styling Implementation (Native CSS / Tailwind)**
*   Use standard CSS variables embedded at the `:root` level for strict aesthetic enforcement.
*   **Mobile-First Construction:** Layouts must be built using CSS Grid/Flexbox targeting mobile dimensions first, applying media queries (`@media (min-width: 768px)`) for desktop scaling.
*   **Hardware Acceleration:** All animations must target GPU-accelerated properties (`transform: translate3d`, `opacity`, `scale`) to avoid layout thrashing on lower-end mobile devices.

## 6. Site Architecture & Navigation Matrix

To balance an iOS-native, ultra-clean aesthetic with aggressive SEO and growth strategies (like our Flighty benchmark), the site uses a **Hub-and-Spoke Flat Hierarchy**. It prevents visual clutter while maintaining a wide SEO footprint.

### 6.1. Page Hierarchy (ASCII Map)

```text
Homepage (/)
├── Product 
│   ├── Pricing (/pricing)
│   ├── Simulator (/simulator)
│   ├── Records & Export (/records)
│   └── Enterprise & Bulk (/business)
├── Compare (SEO Battle Cards)
│   ├── vs. Paper Charts (/compare/paper-charts)
│   └── vs. Maps Anesthesia (/compare/maps-anesthesia)
├── Resources
│   ├── Mission & Security (/security)
│   └── Help & FAQs (/support)
├── Legal 
│   ├── Privacy Policy (/privacy)
│   └── Terms of Service (/terms)
└── Web App Flow (React Islands)
    ├── Sign In (/login)
    └── Dashboard (/app) 
```

### 6.2. Navigation Zones

**Header Navigation (The Mobile-Friendly "Pill")**
*   **Placement:** Fixed/Sticky top-center pill (with `backdrop-filter: blur`).
*   **Items (Max 4):** Logo (Left), Pricing, Simulator, Security.
*   **Primary CTA:** Right-aligned "Sign In" or "Get the App" button.
*   *Note:* The header remains ruthlessly simple. SEO and secondary pages are strictly relegated to the footer.

**Footer Navigation (The 4-Pillar Hub)**
*   **Organization:** Deep categorization to pass link equity without creating visual noise.
    *   *Product:* Pricing, Simulator, Records, Enterprise
    *   *Compare:* vs. Paper Charts, vs. Maps Anesthesia
    *   *Company:* Mission & Security, Support
    *   *Legal:* Privacy, Terms

### 6.3. Information Architecture Rules
*   **The 2-Click Rule:** Users must reach checkout or app login within 2 clicks from any point on the site.
*   **URL Policy:** All URLs must be lowercase, hyphenated, and highly descriptive. Minimize folder nesting to 1-level deep (e.g., `/compare/paper-charts`).
*   **No Orphans (The Footer Safety Net):** Legal pages and all SEO Comparison modules must be internally linked via the Footer to satisfy web crawlers while protecting the absolute minimalism of the main interface.

## 7. AI Agent Execution Protocol
*(Instructions specifically formatted for Autonomous Development Agents like Antigravity, Codex, or Claude.)*

If you are an AI agent tasked with building or updating this website, you **MUST** adhere to the following strict execution protocol to protect the user's budget, maintain performance constraints, and prevent code bloat.

### Phase 1: Context Intake (Do not write code before doing this)
1. Read `/Foundational Documents/Website Design and Aesthetic.md` to map the exact iOS-native typography, border-radii (`24px`-`50px`), and minimalist color palettes into your `<style>` tags or Global CSS tokens.
2. Read `/Foundational Documents/homepage_copy_blueprint.md` so you know exactly what copy belongs in each Astro section.
3. Read `/Foundational Documents/brand_identity_bolus.md` to ensure the tone, voice, and positioning of any generated text strictly align with the established brand protocol.

### Phase 2: Project Initialization (CLI Commands)
When asked to scaffold the project, execute strictly the following stack:
```bash
# 1. Init minimal Astro (No template bloat)
npm create astro@latest ./ -- --template minimal --install --no-git --yes

# 2. Add exact necessary integrations
npx astro add react cloudflare sitemap --yes

# 3. Add Framer Motion for the UI animations
npm install framer-motion
```

### Phase 3: Coding Guardrails
1. **Strict 0kb JS Default:** Every page must remain an `.astro` file using static HTML/CSS. If you create a React `.tsx` file, it must ONLY be for an interactive component (like a payment toggle or simulator).
2. **Islands Architecture Enforcement:** When writing `<Component />` inside Astro, you must append `client:visible` or `client:idle`. Never render a React component without a hydration directive unless it requires $0.00 client-side execution.
3. **No Bloated Libraries:** DO NOT install generic UI libraries (e.g., Material UI, Chakra, or Bootstrap). All styling must be bespoke, mapping to the CSS variables defined by the design document.
4. **Cloudflare Bound:** Ensure the `astro.config.mjs` targets `adapter: cloudflare()` so the deployed output perfectly matches Cloudflare Pages infrastructure.
